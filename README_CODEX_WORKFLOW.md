# README_CODEX_WORKFLOW.md

## 目标

这份文档规定：如何在复杂 C++ 工程里把任务交给 Codex，同时保持：

- 速度
- 可控性
- 低 bug 风险
- 你自己对代码逻辑的理解

---

## 核心原则

不要把 Codex 当作“自动把整个项目写完的人”。  
要把它当作 **受约束的工程代理**。

你自己负责：

1. 系统边界
2. 接口契约
3. 验收标准
4. 最终合并决策

Codex 负责：

- 阅读代码
- 定位问题
- 小范围实现
- 写测试
- 修已复现 bug
- 解释 diff

---

## 推荐任务流

### 1. 先理解，再实现

复杂任务第一轮不要直接写代码，先让 Codex 输出：

- 它理解的目标
- 它计划修改的文件
- 它不会修改的文件
- 实现思路
- 风险点
- 测试方案

### 2. 再做最小实现

要求：

- 最小 patch
- 不做顺手重构
- 不扩大修改范围
- 保持可 review

### 3. 最后强制解释

改完后要求它输出：

- 改了哪些文件
- 每个文件改了什么
- 为什么这样改
- 风险点
- 如何验证

---

## 推荐 agent 分工

### A. 设计总览agent
只看代码，不写代码。负责整体流程的把控
产出：

- 调用链
- 类职责
- 数据流
- 风险点

### B. 实现 agent
只做一个明确功能，不负责架构改造。

### C. 测试 agent
专门补：

- 单测
- 边界 case
- 回归测试
- 最小复现

### D. Review / 解释 agent
不负责写代码，只负责解释这次 patch 值不值得合并。

---

## 不要这样派任务

错误示例：

> 帮我把前处理、推理、显示、线程池都写完。

这种任务范围太大，Codex 很容易误解你的意图并乱改。

---

## 应该这样派任务

正确示例：

> 在 `src/preprocess` 中实现蛇形扫描 patch 迭代器。  
> 输入是单通道 `uint8` 大图。  
> 输出为 512x512，stride=256 的 patch 顺序流。  
> 边缘不足丢弃。  
> 只允许修改：
> - `include/preprocess/patch_scanner.h`
> - `src/preprocess/patch_scanner.cpp`
> - `tests/test_patch_scanner.cpp`
> 不允许修改 runtime / display / CMake 其他模块。  
> 先给实现计划和测试方案，再提交 patch。

---

## 复杂任务的三轮制

### 第一轮：骨架
- 接口
- 空实现
- 编译通过
- 日志打通

### 第二轮：正确性
- 功能实现
- 单元测试
- 集成测试

### 第三轮：质量
- 边界条件
- 性能
- 注释
- 小范围整理

---

## 什么时候不要让 Codex 自动大改

这些场景先让 Codex 做方案分析，不要直接写代码：

- 线程模型调整
- 所有权重构
- 跨模块接口改造
- CMake 大改
- 引入新依赖
- 硬件/驱动/板端交互
- 性能关键路径重写
- 你自己需求还没想清楚的时候

---

## 每个任务的固定检查问题

下任务前你自己先问一遍：

1. 我到底要改什么行为？
2. 影响范围最小能缩到哪？
3. 哪些文件不该动？
4. 完成标准是什么？
5. 怎么验证？
6. 哪些错误是绝对不能引入的？

如果这些问题你自己都答不清，先不要让 Codex 开工。
