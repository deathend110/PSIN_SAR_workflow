# REVIEW_CHECKLIST.md

> 用于人工 Review Codex 产出的 patch。  
> 每次合并前，至少过一遍这份清单。

---

## A. Scope

- [ ] 改动是否严格围绕任务目标？
- [ ] 是否出现“顺手重构”？
- [ ] 是否改了不该改的模块？
- [ ] 是否引入了不必要的新文件或依赖？

---

## B. Interface stability

- [ ] 是否改动了公共接口？
- [ ] 如果改了，是否有充分理由？
- [ ] 是否会影响其他调用方？
- [ ] 是否保持了原有使用习惯？

---

## C. Ownership and lifetime

- [ ] 是否引入了新的裸 owning pointer？
- [ ] 是否把原本简单的所有权变复杂了？
- [ ] 是否无理由引入 `shared_ptr`？
- [ ] 生命周期是否更清楚而不是更模糊？

---

## D. Threading and synchronization

- [ ] 是否改动了线程模型？
- [ ] 是否可能引入 race condition？
- [ ] 是否可能引入死锁？
- [ ] 共享状态是否有明确同步？
- [ ] 是否在不该阻塞的路径中增加了阻塞操作？

---

## E. Correctness

- [ ] 边界条件是否考虑完整？
- [ ] 空输入 / 非法参数是否处理？
- [ ] 索引是否可能越界？
- [ ] 格式转换是否正确？
- [ ] 错误路径是否可达且处理合理？

---

## F. Performance

- [ ] 热路径是否引入多余复制？
- [ ] 是否新增不必要分配？
- [ ] 是否在循环中引入昂贵操作？
- [ ] 是否有明显可避免的性能退化？

---

## G. Testing

- [ ] 是否新增或更新测试？
- [ ] bug 修复是否有回归测试？
- [ ] 测试是否真的覆盖任务目标？
- [ ] 是否只验证“能跑”，而没有验证“结果对”？

---

## H. Explainability

- [ ] 这次改动是否容易理解？
- [ ] 核心逻辑是否比之前更清楚？
- [ ] 注释是否解释了“为什么”，而不是重复“做了什么”？
- [ ] diff 是否适合审查？

---

## I. Merge decision

### 可以直接合并
满足条件：
- 范围小
- 行为正确
- 风险低
- 测试充分

### 需要修改后再合并
常见原因：
- 范围偏大
- 测试不足
- 接口扩散
- 解释不清

### 不应合并
常见原因：
- 设计误解
- 影响面过大
- 高风险线程/所有权修改
- 无法验证正确性
